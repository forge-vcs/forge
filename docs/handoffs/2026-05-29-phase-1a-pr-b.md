## PR B = U3, U4, U5, U6 (see the plan for full per-unit detail)

- **U3 — SQLite connection hardening** (`forge-store::open_connection`): set `journal_mode=WAL`, `synchronous=NORMAL`, and a `busy_timeout` on every connection open. Keep `foreign_keys=ON`.
- **U4 — IMMEDIATE write transactions + bounded busy-retry**: every writer uses `BEGIN IMMEDIATE`; a bounded retry helper absorbs `SQLITE_BUSY`/`SQLITE_BUSY_SNAPSHOT`.
- **U5 — serialize the `--request-id` replay check** so a concurrent same-id retry replays cleanly (exactly one set of domain rows).
- **U6 — concurrency + durability exit-criteria integration tests** (new `crates/forge-cli/tests/forge_concurrency.rs`).

**Exit criteria (from the plan / roadmap):** ≥8 concurrent processes complete the compete loop with zero `SQLITE_BUSY` and zero ID collisions over a 10k-mint loop; a command retried with the same `--request-id` creates exactly one domain row.

## Non-negotiable gotchas the reviews surfaced — honor these (details + rationale in the plan)

1. **13 writer transaction sites, not 11.** Apply IMMEDIATE to *every* `connection.transaction()` that does an INSERT/UPDATE — re-grep `connection.transaction()`; the easily-missed one is **`attach_attempt`**. Implement against the rule, not a fixed list.
2. **`BEGIN IMMEDIATE` does NOT cover separate-connection reads.** `propose`, `record_check`, `record_evidence` read "latest" (`latest_snapshot_for_attempt` / `latest_evidence_for_attempt`) on a *separate connection before* their write txn. Move those determining reads **inside** the IMMEDIATE txn (query on `tx`, as `save_snapshot` already does for its parent pointer). If a read genuinely can't be moved, document it and defer — do **not** claim read-then-write atomicity you didn't deliver.
3. **rusqlite API:** `pragma_update(None, "journal_mode", "WAL")` **errors** (`ExecuteReturnedResults`) — use `execute_batch("PRAGMA journal_mode=WAL;")`. Use `Connection::busy_timeout(Duration)`. Writers: `transaction_with_behavior(TransactionBehavior::Immediate)`.
4. **`SQLITE_BUSY_SNAPSHOT` (517) is NOT retried by `busy_timeout`** — IMMEDIATE avoids the upgrade race; the bounded retry must still catch 517 defensively.
5. **U5: pin option (a)** — re-check `operation_for_request` *inside* the IMMEDIATE txn as the **first statement**, sharing one txn with the domain inserts. The unique-violation-catch variant is a fallback only (not correctness-equivalent). Preserve the command-aware/status-aware replay contract (success→result, failure→failure, different command→`REQUEST_ID_CONFLICT`).
6. **U6 must spawn real OS processes** (`std::process::Command`, driven from threads) — an in-process multi-threaded test sharing one connection does NOT exercise WAL multi-process semantics. Tune retry params before asserting zero-`SQLITE_BUSY`, or restate as "no *unretried* SQLITE_BUSY within the retry budget."
7. **WAL is same-host-only**; `forge.db-wal`/`-shm` sidecars appear (covered by `/.forge/` gitignore + `is_ignored_by_policy`). Document checkpoint-before-copy; full handling is Phase 1b.
