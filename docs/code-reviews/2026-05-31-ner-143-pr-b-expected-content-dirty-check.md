## Summary

PR-B ships the headline NER-143 fix: a worktree dirty-check that is **meaningful for chained navigation** (the "undo twice bricks" P1), built on a crash-safe `current_state.expected_content_ref` baseline (migration 007), plus the undo correctness fixes (R3 cross-attempt binding, R4 `undone_operation_id`). The design — set `expected_content_ref` atomically inside the record txn + an `ensure_clean_worktree` that passes on `worktree == expected OR worktree == target` — was the doc-review-corrected DR-F1 ordering (the naive "write-before-materialize" sketch doesn't self-heal).

**The safety property was the review's focus and it HOLDS.** The adversarial and reliability personas each attacked "can ensure_clean_worktree pass while genuine unsaved work would be clobbered?" and **could not break it**: `matches_target` is safe because materialize-over-target is a no-op over exactly the path set restore touches; `matches_expected` only passes on a saved tree (not unsaved work); the NULL fallback is the stricter latest-snapshot baseline; every error path fail-closes before the clobber. **data-migrations returned clean** (migration 007 additive/nullable/safe; the 6→7 fan-out complete across all 7 pinned sites). **maintainability returned clean** (compliant; the dedicated-UPDATE-not-shared-CAS discipline well-commented).

## Defer-able — Phase 8 / follow-up (noted, not PR-B scope)

- **The during-materialize partial-restore crash state** (reliability residual conf 80). `restore_snapshot` is per-file crash-atomic but the whole-tree restore is a sequence, so a crash PART-WAY through leaves a worktree matching neither expected nor target → next nav refuses `DIRTY_WORKTREE` (the SAFE direction — refuse, never clobber; recoverable via re-run-after-resync or `doctor` reclaiming `.forge-restore-` temps). The DR-F1 crash analysis enumerated crash points (i)-(iii); this fourth point is safe-by-refusal, not self-healing. Documented as a known limitation; no code change needed for safety.
- **A worktree-level advisory lock** to close the materialize-outside-the-DB-lock window for two concurrent forge processes against one worktree (adversarial residual conf 70). Pre-existing (the old code also materialized outside the lock); single-developer-local-loop v0 scope makes it low-likelihood. Track for the multi-process story.

## Reviewed-and-rejected — verified-safe / not reachable

- **Data loss via `matches_target`** (adversarial, verified false). `content_ref` (the snapshot/commit tree) and `restore_snapshot` operate over the exact same path set, so `worktree == target` means re-materialize is a true no-op — no unsaved work to clobber. **Rejected.**
- **NULL-fallback clobber** (adversarial #3, verified false). A pre-007 mid-restore-then-undo: worktree sits on the restored tree A, undo's target is A, so `matches_target` passes and re-materialize is a no-op. No clobber. **Rejected** (safe; the NULL window can spurious-refuse on the FIRST nav, an accepted v0 limitation, but never clobbers).
- **R3 cross-attempt via parent chain** (adversarial #4, verified false). `parent_snapshot_id` is written only by `save_snapshot` from a same-attempt parent query, so binding the latest-snapshot SELECT to the attached attempt cannot cross attempts. **Rejected.**
- **busy/517 retry non-idempotency** (reliability, verified false). The whole closure re-runs on retry; `set_expected_content_ref` is an idempotent absolute UPDATE; the CAS in `insert_operation_view` runs first and aborts the closure before the expected write if the parent op moved — so expected is never set against a moved parent. **Rejected.**
- **No-lock regression** (reliability, verified false). `ensure_clean_worktree` acquires no lock; it relies on the command-boundary advisory lock already held by `command_result` for the mutating nav commands. Acquire-once preserved. **Rejected.**
- **Reconcile interaction** (reliability, verified false). The new write is pure worktree-state; `reconcile_native_head` touches only the ref-store HEAD + commit objects. No interaction with HEAD-lags-never-leads. **Rejected.**
- **Migration fan-out gap** (data-migrations, verified complete). All 7 schema-head pins updated to 7 + the HEAD+1 stamps to 8; no stray `6`-as-head pin remains. **Rejected** (clean).
