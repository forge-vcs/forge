## Reviewed and rejected (intentional / documented — do not re-flag)

- **Tamper-EVIDENT, not tamper-PROOF: a full chain rewrite up to the head by a whole-DB-write attacker is undetectable** (adversarial/security residuals). This is the explicit Phase-5 boundary — `integrity.rs` and the plan state it; cryptographic signing is Phase 9. The op-link folding catches naive, partial, and *interior* tampers (incl. the recompute-row-hash attack at the first un-rewritten downstream op); only a rewrite-to-head evades. Not a defect.
- **The gate path (`check`/`accept`) catches a *naive* evidence edit via the cheap per-row recompute, but not a row whose `content_hash` was also rewritten — that requires `doctor`'s op-walk** (correctness P2 advisory). This is the deliberate two-level design (gate = cheap in-txn per-row; `doctor` = deep offline re-walk), documented in the plan's Key Decisions and the code. The exit criterion ("editing any evidence row → refused on re-evaluation") is met because a real edit leaves the hash stale → the gate refuses. Not a defect.
- **Structured-gate parsing trusts the declared command's honesty (same trust boundary as `exit_code`)** (adversarial/security residuals). A PATH-shimmed or `--no-run` cargo can emit a `0 failures` summary — explicitly scoped out in `parsers.rs` and the plan's Scope Boundaries. A non-passing run cannot be spoofed to `failed:0` (it degrades to `None` → `missing`, fail-closed), so the gate can only be *withheld*, never weakened.
- **Redaction false-negative floors** (security residual): the 7/8/40/64-hex + UUID exemption (to avoid redacting Forge's own SHAs/content hashes), the 20-char entropy length gate, the JSON key allowlist, and the scheme-anchored credential-URL matcher (scp-style untouched) are documented design floors, now surfaced in `notes.secret_protection`. Not defects.
- **`stdout_truncated` reflects raw-output-exceeded-cap, not redacted-excerpt length** (correctness P3). The flag is hashed and self-consistent (verification reads the same stored flag); it means "raw output exceeded `EXCERPT_LIMIT`." Semantic only. No change.
- **`insert_operation_view` thin wrapper over `insert_operation_view_chained`** (maintainability P3). The wrapper keeps the 8 non-domain callers unchanged (passing `None`); inlining would be 8× churn. Deliberate. No change.
- **New serde structs omit explicit `#[serde(rename_all)]`** (project-standards checked): fields are already snake_case, consistent with every existing forge-store struct. `RedactionKind`/`GateVerdict`/`TamperKind` (the multi-word-prone enums) carry it. Correct as-is.

## Deferred to follow-up (filed against NER-136 / future tickets)

- **D1 — Harden the `integrity_marker` against in-chain laundering (defense-in-depth).** The marker row is outside the hash chain; raising `evidence_high_water` (or `UPDATE`-ing a TEXT-PK rowid below it) relabels post-watermark `missing_hash` tampers as grandfathered. Within the conceded whole-DB-write boundary, but a *cheaper* laundering path than a full rewrite. Options: a `doctor` consistency assertion (marker should not exceed the count of NULL-hash rows), or fold the marker values into the genesis/init op link. (security P2 + adversarial P2.)
- **D2 — `accept` defense-in-depth: walk the deciding evidence's op-link in-txn**, so `accept` (which gates export) catches the recompute-row-hash attack without relying on a separate `doctor` run. The plan's "Deferred to Implementation" already flagged the one-link-vs-N choice. (correctness P2.)
- **D3 — `export` verifies the decision row only, not the deciding evidence.** A post-accept evidence tamper before export is caught by `doctor`, not refused at export. The exported branch *content* is from the snapshot `content_ref` (not evidence), so the branch is not corrupted — but the published "passing-check" provenance can rest on post-accept-edited evidence. Consider re-verifying deciding evidence at export. (security residual.)
- **D4 — Residual test coverage** (testing/api-contract/maintainability): `FORGE_ACTOR` env resolution path; assert `warnings[]` populated on a redacting `forge run`; `parse_cargo_clippy` degrade-to-`None` on a compile-error/timeout (no summary, no `Finished`); a different/negative `exit_code` changing the evidence digest (the `-1` timeout path); a multi-process concurrency test that a tampered row is caught by a concurrent in-txn `check`; an `op_domain_digest` redirect-the-view-binding tamper.
- **D5 — Minor maintainability:** dedup the three near-identical `*_high_water` helpers behind a column-kind enum; push `RedactionKind` per-match instead of the count-then-loop pattern in the redactors. (maintainability P2/P3.)
- **D6 — Observability:** `GateResult` does not serialize `structured_failures`, so a consumer cannot distinguish "failed on exit code" from "failed on parsed count" from the `check --json` output. (api-contract residual.)
