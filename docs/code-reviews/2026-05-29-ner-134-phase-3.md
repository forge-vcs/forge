## Defer-able — real, but out of [S] scope; follow-ups

- **C1 / M2 / project-standards (P2) — `restore`'s binding check uses `resolve_attempt(None)`, while `save`'s uses the raw `attached_attempt_id`; and `save` has a named `verify_save_target` store helper while `restore` reconstructs the check inline.**
  Convergent finding (correctness, testing, maintainability, project-standards). **Not a
  reachable contamination hole in v0:** (a) there is no `abandon` command, so the attached
  attempt is always active or `None`, and when `Some & active` `resolve_attempt(None)`
  *equals* the raw `attached_attempt_id`; (b) the pre-existing dirty-worktree check (also
  `resolve_attempt(None)`) already refuses a restore whose target content diverges from the
  bound attempt's latest, so the "relocated clobber" the correctness reviewer constructed is
  caught as `DIRTY_WORKTREE`. The reachable, tested scenario (attach A2, restore A1's
  snapshot → `ATTEMPT_WORKTREE_MISMATCH`) is correct. **Follow-up:** extract a store-level
  `verify_restore_target(cwd, snapshot_id)` that reuses `verify_worktree_binding` against the
  raw binding, unifying the two surfaces' definition of "bound attempt" and removing the
  inline error construction from `main.rs`. File against M2/Phase-3 follow-ups.
  - Note (project-standards): restore's check is **structurally CLI-only** — it must run
    *before* `restore_snapshot` materializes (post-materialization is too late to prevent a
    clobber), so an "authoritative write-path guard" like `save_snapshot`'s is not possible
    for restore. The CLI pre-check is the correct and only place. Not a gap.

- **ACF-001 (api-contract, P2) — `ATTEMPT_WORKTREE_MISMATCH.details.requested_attempt` has different provenance on the two surfaces** (caller's `--attempt` for `save`; the snapshot's owner attempt for `restore`).
  `attached_attempt` is unambiguous on both (always the worktree binding), and the `Display`
  remedy ("attach `{requested_attempt}` first") is valid for both (attaching the snapshot's
  owner then restoring works). Agents branch on `code` + `details`, both well-defined.
  **Follow-up (optional):** add an `originating_command` detail key, or a `forge schema`
  notes entry clarifying restore's `requested_attempt` provenance. Low value; deferred.

- **T2 (testing, P3) — the plan named a dedicated `tests/forge_errors.rs` error-contract test; the assertions live in `forge_attempts.rs` instead.**
  Functionally equivalent coverage exists (code + both detail keys + retryable on the save
  surface; restore surface covered by T1's addition). Deferred — discoverability nicety.

- **T3 (testing, info) — the in-`save_snapshot` authoritative `verify_worktree_binding` guard is only exercised via the CLI path** (the CLI pre-check fires first), so no test reaches it independently.
  Defense-in-depth is intact (the guard exists and can't be bypassed by a non-CLI caller);
  a forge-store unit test calling `save_snapshot` directly with a mismatched context would
  pin it. Deferred — needs DB-fixture setup.

## Defense-in-depth — noted, not blocking

- **Security S1 (residual) — the "no filesystem paths in `anyhow` context" invariant on `current_base`/`base_content_ref` is enforced by convention, not the type system.**
  Verified clean today: both backends' `git()` helpers interpolate only the git argv + git's
  stderr (cwd is passed via `.current_dir()`, never formatted). The trait doc pins S1.
  Follow-up: a test asserting these error strings are path-free would harden it. (Phase 7.)
- **Security S2 (residual) — `base_content_ref`'s secret-exclusion depends on git's tree + `restore_snapshot`'s `is_ignored_by_policy` filter.**
  Verified: attach/restore via the `git-tree:` base ref routes through the git backend's
  policy-filtered `tree_paths`, so `.env`/keys in the base tree are never materialized. The
  trait doc pins S2 for the Phase-7 native walker. Follow-up: a cross-backend round-trip test
  planting a `.env`/`.pem` in the base tree. (Phase 7 / NER-138.)
- **project-standards (residual) — restore's pre-check + materialize + `record_restore` is not one locked transaction (TOCTOU window vs a concurrent `attach`).**
  Same shape as `save`'s documented pre-check pattern; `record_restore` holds the per-command
  advisory lock. A concurrency-harness test is the only way to exercise it. Deferred.

## Reviewed-and-rejected — known, intentional; do not re-flag

- **M1 (maintainability, P2) — "native backend duplicates git base-logic; make `forge-content-native` depend on `forge-content-git`."**
  **Rejected by design.** The plan's resolved decision **D3** (was open-question O3) chose the
  ~3-line shell-out duplication over a new crate dependency: `forge-content-native` already
  shells git directly for `ls-files`/`diff`, the trait crate `forge-content` is git-free by
  design (so a shared helper can't live there), and the duplication vanishes in Phase 7
  (NER-138) when native stops using git. The `// Phase 7` markers document the temporary
  coupling. Keeping the dependency graph acyclic and the Phase-7 deletion surface small
  outweighs deduping 3 lines.
- **"`Display` message is misleading for restore" (api-contract framing).**
  **Rejected.** "run `forge attempt attach {requested_attempt}` first" is valid guidance for
  both surfaces — attaching the requested attempt then re-running works for save and restore.
