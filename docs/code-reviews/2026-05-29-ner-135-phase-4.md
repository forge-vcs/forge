## Real-actionable — fixed in this PR (commit `130af2d`)

- **F1 (correctness P2 + adversarial P2, corroborated by maintainability/data-migrations/learnings) — fail-OPEN on an unparseable `check_spec_json`.** `intent_check_spec` used `serde_json::from_str(&json).ok().unwrap_or_default()`, so a non-NULL but unparseable spec (manual DB edit, partial write, or a future spec shape this binary can't read — the schema_version gate does **not** catch this, since 003 only adds the column) silently collapsed a declared multi-gate intent to permissive default mode, where a lone `run -- true` passes. The one place the enforcement gate could be defeated by data drift now **fails closed**: NULL → default mode; present-but-unparseable → error. (`crates/forge-store/src/lib.rs` `intent_check_spec`.)
- **F2 (security P1 ×2) — secret-like argv leaked through two egress surfaces.** (a) The `check --json` success envelope emitted `data.gates[].program/args` **un-redacted** — a `missing` declared gate copies the spec's argv verbatim, so a secret in a `--require` spec surfaced on the success path (the plan's threat model only covered the error `unmet`). (b) `redact_secret_like_text` keys on the **first** `=`/`:` of its input, so redacting a space-joined identity (`cargo test FOO=bar --token=X`) missed a secret after a non-secret `key=value`. Fixed by redacting **per-token** at the `CHECK_NOT_PASSED.unmet` error surface and **per-arg** on the `check` `gates[]` egress (`redact_gate_result`), so a secret in any argv position is caught. Regression test extended with a multi-`key=value` case. (`crates/forge-store/src/lib.rs` `redact_gate_result` + `record_check`; `crates/forge-store/src/error.rs` `details`.)
- **F3 (testing P1) — declared-gate `stale` verdict had no integration coverage.** Only the default-mode stale path was integration-tested; the declared-gate stale branch was unit-only. Added `declared_gate_is_stale_when_evidence_is_on_an_earlier_snapshot`.
- **F4 (testing + data-migrations P2, corroborated) — convergence test omitted `intents`.** `three_genesis_cases_converge_by_name` checked only `repositories`/`current_state`, so migration 003's `check_spec_json` column was outside the by-name convergence guard. Added `intents` to the table list (case C is stamped at v2, so the runner must apply 003 to it).
- **F5 (data-migrations P3) — concurrency dedup probe lost its meaning.** `forge_concurrency.rs` still counted `version = 2` rows after the HEAD bump; it now probes version 3 — the migration whose `INSERT OR IGNORE` the concurrent-upgrade race actually exercises.
- **F6 (testing P2) — `CHECK_NOT_PASSED.details.unmet` unasserted end-to-end.** The accept-blocked integration test now asserts `details.unmet` names the `cargo test` gate, exercising the populated-and-serialized contract (not just the code).
- **F7 (api-contract P2) — `reject` carried `"check_status": null`.** Added `#[serde(skip_serializing_if = "Option::is_none")]` so the `reject` response shape is unchanged; `accept` carries `check_status` (always `Some`).
- **F8 (maintainability P3) — overclaiming isolation comment.** `check_spec_json_from_requires`'s doc said "the CLI does not depend on forge-policy directly," but `CheckRecord.gates` transitively serializes `forge_policy::GateResult`. Comment corrected to state the CLI doesn't *name* forge_policy types but does serialize them transitively.

## Reviewed and rejected (intentional / documented — do not re-flag)

- **accept's behavioral change without a `schema_version` bump (api-contract P1, owner: release).** Accept now requiring a passing check by default is **the feature** (NER-135 R6, ROADMAP Phase 4 exit criterion "accept requires evidence by default"). It is documented in the plan (R6/R9) and the `--allow-unverified` flag is the deliberate migration path. The contract is additive at the code level (new flag, new error code, new fields); `schema.cli.v0` is retained per the additive-contract discipline. Not a defect.
- **`CHECK_NOT_PASSED` replay stickiness under a reused `--request-id` (adversarial P3, correctness residual).** A blocked accept recorded under request-id R, then fixed, then retried with the **same** R replays the stale failure rather than re-evaluating. This is **consistent with the documented deterministic-failure replay contract** (StaleBase/DirtyWorktree behave identically) and **fails closed** (cannot forge a green). The fix would be to make `CHECK_NOT_PASSED` non-sticky, which weakens the idempotency contract. An agent retrying after a fix should use a fresh `--request-id` (or none). Documented as a known interaction; no change.
- **`GateResult`/`CheckOutcome`/`Gate`/`CheckSpec` lack explicit `#[serde(rename_all = "snake_case")]` (project-standards residual).** Their fields are already snake_case (no serialization difference), and every existing forge-store output struct (`CheckRecord`, `DecisionRecord`, …) omits the attribute too. Adding it only to the new forge-policy structs would be *inconsistent* with the established codebase convention. `GateVerdict` (the one multi-word-prone enum) already carries it. No change.
- **Default-mode trivial-pass / intent-scoped gate (adversarial P3).** An agent can start an ungated intent and pass the default check with `run -- true`. This is the **acknowledged Honesty Note** (the gate is content-bound and un-bypassable-by-*trivial-command* only *within a declared-gate intent*; a repo-default-required-gate is explicitly future work). Documented in the plan Scope Boundaries.
- **Empty `unmet` in default-mode-missing/stale (correctness P3).** When an *ungated* intent's accept is blocked, `unmet` is empty (no declared gates) — the `status` field (`missing`/`stale`) carries the signal. Cosmetic; the declared-gate path (the one with actionable gate identities) populates `unmet` and is asserted (F6). No change.

- **Unit tests for `check_spec_json_from_requires` tokenizer + `representative_evidence_id` (testing gaps).** Exercised indirectly; add focused unit tests for faster regression detection. Defer.
