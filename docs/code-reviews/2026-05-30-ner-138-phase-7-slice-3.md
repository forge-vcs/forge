## Defer-able — filed as a Linear follow-up (NER-143)

These are real but are either partly pre-existing, safe-by-refusal (not data-loss), within slice-3's documented v0 scope cuts, or need design — so they are tracked rather than rushed into this PR.

- **`undo`/`checkout` permanent-dirty-worktree chaining (P1, adversarial conf 90)** — after `undo` restores snapshot A, the latest snapshot is still B, so the next `undo`/`checkout`/`restore` dirty-check (worktree-vs-latest-snapshot) spuriously fails `DIRTY_WORKTREE`; "undo twice" is impossible without an intervening `save`. **Root is the pre-existing `restore` dirty-check baseline** (compares against latest snapshot, not the expected current worktree state), now inherited by the new commands. Safe (refuses, never clobbers). A proper fix tracks "expected current worktree content" in `current_state` separately from "latest snapshot" — a design change beyond slice-3 scope. The plan already cut multi-step undo/redo to v0-deferred.
- **`undo` semantics divergence + misleading `undone_operation_id` (P2, correctness conf 70)** — `undo` uses the snapshot `parent_snapshot_id` chain (undo-the-last-save), not the op-log `current_state` rewind R7/U8 described; after a non-save head op (accept/checkout/run) it restores the prior save and labels `undone_operation_id` as the op-log head. The snapshot-chain choice was deliberate (robust, no cross-table timestamp race) and the gc-root invariant holds; the audit-field/semantics alignment is the follow-up.
- **`undo` cross-attempt scope (P2, adversarial conf 60)** — `undo_target` selects the repo-wide latest snapshot while the dirty-check resolves the attached attempt's latest; in a multi-attempt repo `undo` could restore attempt X's content into attempt Y's worktree. `restore` guards this with a `snapshot_owner_attempt_id == bound_attempt` check; `undo` should adopt the same binding. (v0 is single-attempt-focused.)
- **checkout/undo materialize-before-op-log-record ordering (P2, reliability conf 75)** — `restore_snapshot` clobbers the worktree, then `record_checkout`/`record_undo` runs in a separate txn; a crash or `CurrentStateChanged` CAS-loss between them leaves the worktree materialized with no op-log record. Materialize is idempotent so the residual is bounded; the fix is an op-log "intent" row before materialize.
- **gc swallows malformed view `state_json` (P3, reliability conf 75)** — `gc_dry_run`'s op-log-root scan drops an unparseable `views.state_json` row silently, under-counting roots. Dry-run-only today; **latent for Phase 8 mark-sweep deletion** — must fail-closed (conservative) before deletion lands.
- **R15 re-capture-side symlink validation (P2, security/testing)** — the plan's R15 specified validating symlink targets at BOTH materialize AND re-capture; only materialize (the real security boundary) is implemented. See "Reviewed-and-rejected" for why the capture-side gap is not a live leak; filed so the defense-in-depth control + an end-to-end escaping-symlink round-trip test are added.
- **Dirty-worktree precheck triplicated (P2, maintainability conf 75)** — the byte-identical 7-line block in `restore_response`/`checkout_response`/`undo_response` should be a single `ensure_clean_worktree` helper so the "refuse-before-materialize" safety invariant has one definition.
- **Maintainability dedup (P3)** — the redundant `decisions.commit_id` query in `gc_dry_run` vs `native_tip`, and the DanglingCommitId/DanglingParent tip-classification idiom duplicated across `reconcile_native_head`/`native_log`/`verify_native_history`, could share small helpers (the walkers legitimately differ in raise-vs-report policy).
- **e2e git-removed block omits `restore`/`checkout`/`undo` (P3, testing)** — the shell e2e proves the core lifecycle git-free; the Rust integration tests (`native_lifecycle_runs_with_git_removed_from_path`, `native_undo_runs_with_git_removed_from_path`) already cover restore/checkout/undo git-free, so the exit criterion is met — extend the shell block for parity.

## Defense-in-depth — noted, not blocking

- **Pre-existing S1 path-in-context in `materialize_tree`/`sync_dir` (P2, security conf 50, `pre_existing: true`)** — `path.display()`/`full.display()` in `.with_context(...)` reach the untyped `COMMAND_FAILED` envelope `message` on an IO/fsync failure. Present at base `8aea21d` (already reachable via `restore`); slice 3 adds new reachable callers (checkout/undo/set_head). Leaks only the repo's own `.forge`/worktree paths (not secret content like `.env`) and only on an IO failure. Tracked with the S1-hardening surface (replace with the path-free `io::ErrorKind` pattern already used by `read_head`/`map_walk_error`).
- **Fork-induced reconcile brick (adversarial conf 55)** — if a future/alternate `decide()` caller bypassed the CLI stale-base precheck and persisted a forked ledger tip, `reconcile_native_head` would `NativeHistoryCorrupt` on every subsequent lock-holding command. Not reachable via the current single CLI entrypoint (the precheck + advisory lock serialize accepts); the defensive hardening is to move the base==tip validation into `decide()`'s txn. Flagged for Phase 8 when the merge engine introduces multi-parent commits (which also breaks reconcile's first-parent-only walk).
- **Native init in a subdirectory creates a nested `.forge` (adversarial conf 55, P3)** — native `init` anchors root at `cwd` without an ancestor-`.forge` guard (unlike git init's toplevel snap); `forge_root`'s nearest-ancestor walk then routes to whichever repo is closer up-tree. Edge case; an ancestor-`.forge` guard on native init is the hardening.
- **`export branch` `STALE_BASE.expected_head` semantic shift (api-contract conf 50, P3)** — for a native accept it now carries the accepted `commit_id` (correct post-commit-on-accept) instead of `base_head`. Correct and internally consistent; native export is a young surface. A schema-doc/CHANGELOG note is the only follow-up.
- **Migration advisories (data-migrations, all P3, all confirmed safe)** — the `NOT NULL DEFAULT 1` ADD COLUMN is safe via the default; the convergence guard omits `decisions` (stub-table fixture limitation, covered by `behind_db_upgrades_to_head` on the real baseline); the two `native_object_format` version columns are write-only forward-compat markers (unenforced until Phase 9 reads them). No action required.

## Reviewed-and-rejected — false positives / verified-safe / intentional

- **Object-kind header ambiguity on a magic-prefixed legacy blob** (adversarial) — a legacy raw-payload blob whose content starts with `b"forge-object\n"` is correctly disambiguated by the `hash(file) == id` check in `read_object`/`all_object_ids` (the legacy branch re-hashes the raw bytes); ambiguity is **hash-resolved, not format-guessed**. Verified safe + covered by `object_kind_headers_coexist_with_legacy_objects`. **Rejected.**
- **Symlink-chain / write-through-symlink escape** (adversarial/security) — `materialize` never writes a file *through* a symlink: every tree entry is materialized at its own explicit `rel` path, `validate_tree_entry` forbids `/`/`..` in names, and escaping targets are rejected at creation. A symlink leaf cannot have tree children. **Rejected** (structurally not reachable).
- **R15 re-capture content leak** (security) — `scan_worktree` captures a symlink via `read_link` (the target STRING only, never followed), so an escaping/absolute symlink leaks no pointed-at content into a snapshot. The materialize-side rejection is the real security boundary. The missing capture-side *validation* is defense-in-depth (filed above), **not a live leak** — and capture-side rejection would break `forge save` on a repo with a legitimate absolute symlink. **Rejected as a leak; deferred as defense-in-depth.**
- **`Hex64` newtype is premature abstraction** (maintainability raised, then withdrawn) — it is genuinely consumed at the `decide()` commit-build boundary where it makes excerpt text *structurally unrepresentable* in `evidence_digest` (a release-build, type-level guarantee a `debug_assert` cannot give). The plan's Open-Questions weighed this and chose the newtype for the security-sensitive field. **Rejected** (the reviewer concurred on second look).
- **`reconcile_native_head` re-acquires the lock / lazy-write-without-lock** (learnings) — verified: reconcile runs from `command_result`'s `requires_repo_lock` branch (lock already held), opens its own read connection + ref-store writes but never calls `acquire_repo_lock`; the acquire-once-never-nested invariant holds. **Rejected** (verified-safe, flagged as the verification point).
- **`git_root` orphaned by `forge_root`** (maintainability) — `git_root` stays live for git-backend `init` root resolution; not dead code. The triple-hash legacy fallback stays reachable + tested (prove-before-delete). **Rejected.**
