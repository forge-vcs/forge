## Real-actionable — fixed in this review pass

- **A (correctness P2 + adversarial P1 + reliability P2, corroborated ×3) — `diff_trees` lost rename counts and could leak a C-quoted secret filename.** `git diff --name-status`/`--numstat` (a) key renames as `old => new` (config-dependent on `diff.renames`), so a renamed file's insertions/deletions silently dropped to `None`, and (b) C-quote any path containing a tab/newline/non-ASCII byte (e.g. `.env\ttest` → `".env\ttest"`), which `is_secret_risk_path` then fails to match — leaking a secret filename into the `compare --diff` output. Fixed by parsing both passes with `-z --no-renames` (NUL-delimited, never quoted; renames become keyed A+D pairs). Added `diff_trees_keeps_counts_for_a_rename` and `diff_trees_drops_a_secret_path_with_non_ascii_bytes` tests. (`crates/forge-export-git/src/lib.rs`.) *(The identical pre-existing flaw in `filter_secret_paths_from_tree`'s `ls-tree --name-only` export path is **deferred** — out of this PR's scope; see below.)*
- **B (reliability P1, conf 100) — hollow provenance trailer on a missing/non-accepted decision.** `build_publication_trailer` (called by `verify-branch` independently of `export`, which gates on `accepted`) folded an empty decision digest when no decision row existed, so a manufactured commit referencing a never-accepted revision produced a self-consistent digest that `verify-branch` would confirm as `verified`. Fixed: `decision_digest_and_actor` now errors `NotAccepted` when the revision has no decision row or its latest decision is not `accepted` — fail-closed at both export (defense-in-depth) and verify-branch. (`crates/forge-store/src/lib.rs`.)
- **C (agent-native P1, conf 100 + api-contract P3) — `verify-branch` on a non-Forge commit returned `COMMAND_FAILED`.** An agent using `verify-branch` as a CI gate could not distinguish "this branch was not produced by Forge" from a generic failure. Added a typed `MISSING_PROVENANCE_TRAILER { branch, missing_field }` with the full additive drift-guard fan-out (enum, `code()`, `details()`, `Display`, `error_registry()` `ErrorCodeSpec`, both `error.rs` drift-guard tests, `missing_provenance_trailer_details_carry_only_ids`, and `FORGE_ERROR_CODES` — 22 → 23 codes). `verify_publication_trailer` returns it instead of a bare `anyhow!`. Added `verify_branch_on_a_non_forge_commit_is_a_typed_missing_trailer_error`. (`crates/forge-store/src/error.rs`, `crates/forge-export-git/src/lib.rs`, `crates/forge-cli/tests/{forge_schema.rs,forge_accept_export.rs}`.)
- **D (maintainability P1, conf 100) — `gate_identity` was byte-identical to `forge_policy::identity_string`.** Exposed `forge_policy::identity_string` as `pub` and deleted the forge-store copy (forge-store already depends on forge-policy), so the gate-identity rendering in the commit trailer and in `CHECK_NOT_PASSED.unmet` can never silently diverge. (`crates/forge-policy/src/lib.rs`, `crates/forge-store/src/lib.rs`.)
- **E (agent-native P2 + adversarial residual) — a `legacy_unverified` attempt was rankable with no signal in `rank_reason`.** A pre-Phase-5 (grandfathered) deciding row is rankable (only `tampered` is excluded), so an agent selecting rank-1 could accept never-hash-verified evidence. The `integrity` field already carried the label; `rank_reason` now also appends a `legacy_unverified` caveat so a rank-only consumer sees it. (`crates/forge-store/src/lib.rs`.) *(On a fresh Phase-5+ repo `evidence_high_water = 0`, so any NULL hash is `tampered`, not legacy — this only affects repos with pre-Phase-5 evidence.)*
- **F (agent-native P2, conf 75) — `compare --diff` hunk truncation was not surfaced in `warnings[]`.** `FileDiff.truncated` was only at the per-file level; an agent checking the envelope `warnings[]` missed it. `compare_response` now appends one warning per truncated file (consistent with the existing `secret_export_warnings` pattern). (`crates/forge-cli/src/main.rs`.)

## Reviewed and rejected (intentional / documented — do not re-flag)

- **Verify-then-rank uses the *cheap* per-row check; a fully re-hashed row evades it (only `doctor`'s op-walk catches it).** This is the explicit Phase-5 cheap-gate / deep-doctor boundary, honestly scoped in the plan's **Integrity Scope** note, R4/R7/R8, and `schema notes.provenance`. The adversarial and learnings reviewers both **credited** the honest scoping. Adding a doctor-grade op-walk to compare/accept is Phase 5 **D2**, deliberately not pulled in. Not a defect.
- **`verify-branch` PASS is local-consistency, not authenticity (rewrite-ledger-AND-re-export still matches).** The conceded LOCAL-only boundary, documented in `notes.provenance` and the function doc; cross-machine authenticity + signing are Phase 9. Not a defect (security + adversarial confirmed honestly scoped).
- **`secret_export_warnings` surfaces the dropped secret *filename* in a warning string.** Pre-existing, intentional model (the file *content* is dropped from the tree and no hunk is read); matches `export branch` / `pr-body` behavior. Not a leak of content.
- **`ComparedProposal` is a 5-field subset of `ProposalSummary`** (maintainability P2). Deliberate — it is the compare-specific JSON shape (omits `attempt_id`/`changed_paths` that don't belong on a compare row). Keeping a parallel struct is cheaper than serde adapters on the shared type. No change.
- **New serde structs omit explicit `#[serde(rename_all)]`** (project-standards checked): fields are already snake_case, consistent with every existing forge-store struct. Correct as-is.
- **Commit subject changed to `Forge accepted proposal <proposal_id>`; pr-body gains a `## Competing Attempts` section** (api-contract P2 advisory). The `--json` envelope shape is unchanged; consumers key on the `Forge-*` trailer lines / `## ` section names, not subject position. The schema descriptions were updated. Documented behavior change, not a contract break.

## Deferred to follow-up (filed against NER-137 / future tickets)

- **D1 — the pre-existing `filter_secret_paths_from_tree` (export path) has the same C-quoted-filename secret-drop flaw** that fix A closed in `diff_trees`. It uses `git ls-tree -r --name-only` (quotes special-char paths) without `-z`. Out of this PR's scope (pre-existing, on the `export branch` path, not introduced by Phase 6). Fix with `-z` + NUL parsing in a follow-up. (adversarial P1 noted as pre-existing.)
- **D2 — per-attempt connection reuse in `compare_attempts`/`build_compare_row`.** `latest_proposal_for_attempt` / `latest_decision_for_proposal_revision` / `latest_publication_for_proposal_revision` each open their own SQLite connection (3 extra per attempt) rather than reusing the shared `conn`; `pr_body_for` re-opens the repo via `compare_attempts`. At v0 single-user scale this is latency noise and matches the pre-existing `proposal_metadata_for_attempt` pattern; threading the shared `conn` also closes a (theoretical) cross-snapshot consistency window. Performance P2/P3 — defer until a batch/fleet workflow makes attempt counts large. Add a `LIMIT`/scoping guard for the no-selector path then.
- **D3 — structural dedup:** extract a shared `AttemptRecord` row-mapper + `ATTEMPT_SELECT` const (4 copies, the 4th added here); extract a `proposal_by_revision_id` helper to share the `ProposalSummary` column list/mapper. Pre-existing inline-mapper pattern; cosmetic. (maintainability P2/P3.)
- **D4 — residual test coverage** (testing/api-contract/agent-native): a `compare_structured_metrics` unit test (multi-binary summing + clippy_findings + all-None); integration tests producing real `legacy_unverified` / `no_evidence` integrity labels via `aggregate_integrity` (currently only the synthetic `rank_compare_rows` units + e2e exercise them); a `parse_forge_trailers` unit test (prose-before-trailer, missing fields, last-write-wins); a `diff_trees` binary-file test; the `compare --attempt` selector path; `render_trailer_message` empty-intent branch.
- **D5 — `IntegrityStatus` label is a free-form `String` (`verified`/`legacy_unverified`/`tampered`/`no_evidence`) on `AttemptCompareRow.integrity`.** A closed enum with `#[serde(rename_all)]` would make a new variant a compile-then-test failure across the CLI/tests rather than a silent string mismatch (mirrors the Phase 5 `TamperKind` parity pattern). (maintainability residual.)
